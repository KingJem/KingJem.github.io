MobaXterm RDP 经 SOCKS5 代理转发（gost + WSL）

背景

一批 RDP 服务器（107.150.x.x、152.32.x.x 等，端口 3389）只能通过本地的
SOCKS5/HTTP 代理 127.0.0.1:7890（Clash/mihomo 混合端口）访问。目标是让
MobaXterm 里的这些 RDP 会话走这个代理，并且能把其中一台分享给局域网内的同事使用。

关键结论：MobaXterm 的 RDP 不支持 SOCKS 代理

查证 .mxtsessions 格式与 MobaXterm 功能后确认：

  • SSH 会话支持 SOCKS 代理（v6.6 起）
  • VNC 内置查看器支持 SOCKS4/5、HTTP 代理
  • RDP 会话只支持 SSH 网关（跳板机）或微软 RDP/TS Gateway，没有 SOCKS 代理字段

所以不能在 RDP 会话里直接“勾选走代理”。代理必须在 RDP 客户端外部完成——这正是
gost 做端口转发的用武之地。

参考：.mxtsessions 格式（Ruzgfpegk gist）、MobaXterm 文档

方案：gost 本地端口转发

为每台目标分配一个本地端口，gost 监听该端口并把流量经 SOCKS5 代理转发到真实
host:3389。RDP 客户端只连本地端口即可。

为什么放在 WSL 里跑

WSL2 开启**镜像网络（mirrored）**后与 Windows 共享 localhost：WSL 内监听的
127.0.0.1:1339x 端口，Windows 上的 MobaXterm 直接连 127.0.0.1:1339x 就能命中，
无需改会话文件；同时 WSL 内访问 127.0.0.1:7890 也能直达 Windows 侧的 Clash。

同事连不上排查清单

按顺序排查（多数情况是第 1 条防火墙没加）：

  1. 防火墙规则没加/被拦：用管理员跑上面的 New-NetFirewallRule；公司网络可能还有
    组策略防火墙。验证：同事执行 Test-NetConnection 172.16.5.6 -Port 13399，
    TcpTestSucceeded 应为 True。
  2. 不在同一局域网/子网：172.16.5.6 跨网段/经路由器不可达；确认同事与本机同一 LAN。
  3. IP 变了：WLAN 是 DHCP，重连 WiFi/重启路由后 IP 可能改变，重新查 Get-NetIPAddress。
  4. 服务没跑/端口没监听：WSL 内 systemctl is-active rdp-proxy 与
    ss -ltn | grep 13399。
  5. Clash 没开：127.0.0.1:7890 不通则转发链路断；
    curl -x socks5h://127.0.0.1:7890 https://www.google.com 自检。
  6. 镜像网络入站：确认 /etc/wsl.conf 的 [network] networkingMode=mirrored；
    非镜像（NAT）模式下局域网无法经 127.0.0.1 命中 WSL 端口，需改用 Windows 侧
    netsh portproxy 或 hnsdiag/端口代理转发。
  7. 目标本身不可达：本机 curl -x socks5h://127.0.0.1:7890 http://107.150.100.161:3389
    返回 (52) Empty reply 表示 TCP 已通（RDP 不是 HTTP，空回复属正常）；返回连接拒绝
    则代理到目标这一段有问题。
  8. Windows 已占用 13399：netstat -ano | findstr 13399 看是否被别的进程占了。
